1. Introducción y Alcance

1.1 Propósito

El presente documento describe el diseño técnico del Portal de Servicios al Cliente para Chile y Bolivia de Hapag-Lloyd. Este portal permite a clientes y agentes de aduana realizar consultas de Bills of Lading (BL), generar recibos de pago, procesar pagos de flete, cargos locales, demurrage, cambios de bodega y órdenes de servicio (ODS), optimizando los procesos operativos y reduciendo la carga manual del equipo de Customer Service.

Objetivo principal: Digitalizar y automatizar los procesos de pago y consulta de servicios marítimos para las operaciones de Hapag-Lloyd en Chile y Bolivia, reduciendo tiempos de respuesta y mejorando la experiencia del cliente.

2. Contexto de Negocio

2.1 Hapag-Lloyd en Chile y Bolivia

Hapag-Lloyd opera servicios de transporte marítimo de contenedores en Chile y Bolivia. Chile cuenta con una extensa costa pacífica con puertos principales como Valparaíso, San Antonio, Arica e Iquique. Bolivia, al ser un país mediterráneo (sin acceso al mar), depende de los puertos chilenos —principalmente Arica e Iquique— para sus operaciones de comercio exterior, amparado en el Tratado de Paz y Amistad de 1904 que garantiza el libre tránsito de mercancías bolivianas por territorio chileno.

Esta dualidad geográfica genera diferencias operativas significativas que el portal debe manejar:

6. Casos de Uso Detallados

CU-01: Registrar Cliente

Actor: Cliente / Agente de Aduana Prioridad: Alta Precondiciones: El usuario no tiene cuenta en el portal Postcondiciones: Cuenta creada y email de confirmación enviado
Escenario Principal:
  1. El usuario accede a la página de registro del portal.
  2. El sistema muestra el formulario de registro con selección de tipo de usuario (Cliente / Agente de Aduana).
  3. El usuario selecciona su tipo y país (Chile / Bolivia).
  4. El sistema muestra el campo de identificación tributaria correspondiente (RUT para Chile, NIT para Bolivia).
  5. El usuario completa los datos: nombre, identificación tributaria, email, teléfono y contraseña.

14.3 Base de Datos — Creación y Migraciones

El proyecto utiliza Entity Framework Core con un proyecto de migraciones separado (HapagPortal.DatabaseMigrations). La base de datos se crea automáticamente al ejecutar las migraciones.

Paso 1: Verificar SQL Server

Asegúrese de que SQL Server esté corriendo en localhost con autenticación Windows (Trusted Connection). Puede verificarlo con:

sqlcmd -S localhost -E -Q "SELECT @@VERSION"

Paso 2: Configurar Connection String

La cadena de conexión se encuentra en los siguientes archivos:

ArchivoAmbienteUbicación
appsettings.Development.jsonDesarrollobackend/src/HapagPortal.WebApi/
appsettings.Staging.jsonStagingbackend/src/HapagPortal.WebApi/
appsettings.jsonProducciónbackend/src/HapagPortal.WebApi/
appsettings.jsonMigracionesbackend/src/HapagPortal.DatabaseMigrations/

Formato de la cadena de conexión:

Server=localhost;Database=HapagPortalDb;Trusted_Connection=true;TrustServerCertificate=true;MultipleActiveResultSets=true

Importante: Para ambientes de staging y producción, reemplace localhost por el servidor correspondiente y configure credenciales seguras. Nunca use Trusted_Connection=true en producción si el API corre en un servidor diferente al de la BD.

Paso 3: Ejecutar Migraciones

# Desde la raíz del backend cd backend # Opción A: Ejecutar el proyecto de migraciones (recomendado) dotnet run --project src/HapagPortal.DatabaseMigrations # Opción B: Usar EF Core CLI directamente dotnet ef database update --project src/HapagPortal.Infrastructure --startup-project src/HapagPortal.WebApi # Para crear una nueva migración dotnet ef migrations add NombreMigracion --project src/HapagPortal.Infrastructure --startup-project src/HapagPortal.WebApi
